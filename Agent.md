# Agent.md

## 项目定位

Enjoy Agent 是一个以后端为主的 AI Agent 平台学习项目，目标不是快速堆出一个“看起来功能很多”的 SaaS 外壳，而是通过真实代码把 Agent 平台最关键的后端设计吃透。

它重点关注的是这些问题：

- 多租户 Agent 平台怎么组织用户、租户、资产和权限
- RAG 怎样真正接进聊天运行时
- MCP 工具怎样从“登记信息”变成“可运行能力”
- BYOK 和平台官方模型怎样共存
- 计费怎样从简单扣费演进到异步 token 结算
- 市场怎样从“分享对象”演进到“可安装包”

项目当前以后端主链为中心，前端不是当前主要复杂度来源。

## 当前架构主线

当前后端已经形成了比较完整的模块分层：

- `auth / tenant`
- `credential`
- `model / modelgateway`
- `agent / chat`
- `knowledge`
- `mcp`
- `billing`
- `market`
- `shared`

主链路可以概括成：

`用户 -> 凭证 -> 模型 -> Agent -> 聊天 -> RAG -> MCP -> 官方模型 -> 钱包计费 -> 市场安装`

## 当前已经完成的关键能力

### 1. 多租户与管理员边界

- 用户注册、登录、JWT 鉴权
- 租户隔离
- 平台管理员 `systemRole`
- `/api/admin/**` 管理端权限边界

这里的原则很明确：

- 业务资产主要挂在租户下
- 平台运营能力走管理员身份
- 钱包挂用户，不挂租户

### 2. 模型体系

当前模型体系分两条线：

- 用户自己的模型配置
- 平台官方模型配置

关键设计点：

- 运行时显式区分 `USER` / `PLATFORM`
- 官方模型不再混进用户模型表语义里
- Agent 可以绑定用户模型，也可以绑定官方模型

这让日志、计费、权限和市场安装都更稳定。

### 3. RAG 体系

RAG 目前已经不是简单上传文件做检索，而是完整接入 Agent 对话运行时：

- 文档进入 MinIO
- 文本抽取
- chunk 切片
- embedding 入库
- pgvector 向量检索
- 可选 Elasticsearch 混合检索
- Query Rewrite
- Rerank
- 检索调试信息回传

这部分是项目最值得学习的一段，因为它覆盖了从存储到检索到回答注入的完整闭环。

### 4. MCP 体系

MCP 部分已经具备比较完整的平台能力：

- MCP Server 注册与管理
- Tool 目录同步与覆盖
- Agent 绑定 Tool
- 聊天运行时注入 Tool
- Tool 调用日志
- OAuth Auth Code + PKCE
- OAuth Client Credentials
- Token refresh

这意味着项目不仅“支持 MCP”，而是已经把 MCP 真正接成了 Agent Runtime 的一部分。

### 5. 计费体系

计费已经完成从旧方案到新方案的重构：

- 钱包挂在 `user`
- 充值走 `recharge_order -> admin approve`
- 官方模型和价格由管理员维护
- 官方模型调用后记录 `billing_usage_event`
- RabbitMQ 异步消费计费事件
- 按输入/输出 token 计费
- 写入 `user_wallet_transaction`

这个设计比“同步按次扣费”更符合平台语义，也更便于后续扩展。

### 6. 共享市场

共享市场当前已经形成后端闭环：

- 用户提交 Agent / Knowledge Base / MCP Server
- 管理员审核通过、驳回、下架
- 市场展示已审核资产
- 用户安装到自己的租户

当前最核心的设计原则是：

- 市场安装优先复制本地副本
- 不跨租户直接共享运行态对象
- 可共享内容尽量随包复制
- 私密 credential / OAuth 连接态不复制

Agent 安装时目前已经支持：

- 复制 Agent 配置
- 复制知识库文档到目标租户
- 复制 MCP Server 和 Tool 定义
- 自动恢复 Agent 的 MCP Tool 绑定

## 关键设计取舍

### 1. 钱包挂用户，不挂租户

原因很直接：

- 充值审核面对的是平台用户
- 用户可以加入多个租户
- 官方模型扣费主体更像平台用户，而不是某个租户容器

所以租户负责业务资产，用户负责钱包和平台计费。

### 2. 官方模型独立建模，不混入用户模型

如果把官方模型继续塞在用户模型配置里，后面会出现很多语义问题：

- 到底是平台资产还是租户资产
- 价格归谁维护
- 市场安装后怎么判断计费主体

因此现在官方模型、官方 key、价格都放在平台侧独立维护。

### 3. 市场安装生成副本，不做弱引用

项目明确不把市场做成“引用别人对象”的系统。

原因包括：

- 跨租户引用会把删除、权限、配置变更全部搅乱
- MCP 背后可能挂私密 credential 和 OAuth 连接
- 知识库如果引用源对象，用户体验会很脆弱

所以市场更像安装包系统，而不是共享引用系统。

### 4. 计费异步化，但保留数据库事件真相

RabbitMQ 负责异步分发，不承担“唯一真相存储”。

因此计费流程里保留了 `billing_usage_event`：

- 记录价格快照
- 记录 token 快照
- 便于幂等处理
- 便于失败排查与补偿

这条原则后续也适合复制到长任务、异步安装等链路里。

## 当前边界

这些边界是当前版本刻意保留的，不是遗漏：

- 前端管理后台还没有完全收口
- 知识库市场安装当前仍走同步导入，文档多时会慢
- 市场还没有版本升级、评分、收藏、搜索这类运营能力
- 长任务、计划步骤记录还没有正式接入运行时
- 安装流程的异常补偿和重试仍可继续加强
- 还没有进入完整生产化治理阶段

## 当前最值得继续推进的方向

下一阶段我更建议优先补“工程化与稳定性”，而不是再快速扩新模块：

1. 正式整理集成测试脚本和联调流程
2. 补管理端统计、审计和查询接口
3. 把市场安装里的慢操作异步化，并补补偿机制
4. 再进入计划步骤记录、长任务状态这些更强 Agent Runtime 能力

## 开发协作原则

- 先沿现有应用服务主链扩展，不轻易拆散核心编排逻辑
- Controller 保持轻量，业务编排放在 application service
- DTO、实体、仓储继续分层
- 所有表结构修改都必须补 Flyway migration
- 重要新能力优先明确业务语义，再开始写代码
- 文档要随着架构变化同步更新，避免 `README`、`Agent.md`、`Plan.md` 失效
