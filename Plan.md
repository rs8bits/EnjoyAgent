# Plan.md

## 文件用途

本文件记录当前项目已经完成到哪里、当前判断项目处于什么阶段，以及接下来最值得继续推进的顺序。

这份计划已经不再是“怎么把最初 MVP 拼出来”，而是“在已经形成主链闭环的前提下，怎么继续把后端做稳、做深、做成更有代表性的 Agent 平台项目”。

## 当前项目判断

从后端视角看，Enjoy Agent 已经完成了第一版核心业务闭环：

- 认证与多租户
- 凭证与模型
- Agent 与聊天
- RAG
- MCP 与 OAuth
- 官方模型与管理员体系
- 用户钱包、充值审核、异步计费
- 共享市场提交、审核、安装

也就是说，当前阶段已经不再是“有没有主链路”的问题，而是：

- 这条主链路能不能继续做稳
- 哪些地方该进入工程化收尾
- 哪些能力最值得作为下一轮学习与演进重点

## 当前最重要的已完成能力总结

如果只用一句话概括当前版本，可以理解为：

`一个已经具备真实后端闭环的多租户 Agent 平台第一版`

尤其是下面几条主线已经比较成形：

- RAG 真正接入聊天运行时
- MCP 真正接入 Agent Runtime
- 官方模型和用户模型双轨运行
- 计费走用户钱包 + 异步 token 结算
- 市场走“审核发布 + 本地副本安装”

## 接下来最推荐的推进顺序

当前阶段我不建议立刻再铺很多新业务模块，而更建议按下面顺序推进。

### 阶段 13：工程化收尾与稳定性增强

优先级最高。

建议完成：

- 把当前联调脚本整理进仓库
- 固化一套可重复执行的集成测试流程
- 补部署说明、环境变量说明、联调说明
- 梳理关键异常路径和恢复策略
- 补日志与审计视角的检查点

完成标准：

- 新环境可以按文档快速拉起并验证主链
- 核心链路不再依赖手工摸索
- 出问题时更容易定位是模型、检索、MCP 还是计费链路

### 阶段 14：市场安装异步化与补偿

这是最值得继续优化的业务链。

建议完成：

- 知识库文档导入改成异步任务
- 安装进度状态记录
- 安装失败后的可恢复状态
- 部分成功场景的补偿策略
- 市场安装结果可查询

完成标准：

- 大文档安装不阻塞请求
- 用户能看到安装进行中、成功、失败
- 安装失败时平台状态不混乱

### 阶段 15：管理端统计与审计接口

当前后台已经有管理动作，但统计与查询还不够。

建议完成：

- 待审核市场资产数量
- 充值单统计
- 钱包流水查询增强
- 官方模型使用量统计
- 计费事件状态统计
- 市场安装量、上架量、下架量统计

完成标准：

- 管理后台不只“能操作”，还能看全局状态
- 审核、计费、市场安装都有查询和追踪依据

### 阶段 16：计划步骤记录与长任务状态

这是下一轮最值得继续增强的 Agent Runtime 能力。

建议完成：

- 计划步骤实体
- 长任务实体与状态机
- 步骤执行记录
- 与聊天会话或 Agent 任务关联
- 为后续调度和自动化执行预留结构

完成标准：

- 系统可以表达“Agent 正在执行一项持续任务”
- 任务不是只有一段聊天记录，而是有步骤、有状态、有进度

## 当前明确不作为最高优先级的事情

下面这些事情不是不做，而是现在不该抢占前面几项的优先级：

- 很重的前端后台重构
- 复杂 RBAC
- 一开始就上完整风控系统
- 多 provider 超大统一抽象
- 市场评分、评论、收藏等偏运营功能
- 安装包版本升级和复杂迁移系统

## 当前执行原则

- 优先保证主链路真实可跑
- 优先补能明显提升稳定性和可验证性的能力
- 新能力优先走完整纵向链路，而不是只铺表结构
- 保持多租户、凭证加密、迁移、日志、审计这些基本纪律

## 现在如果立刻继续开发，推荐顺序是

1. 工程化收尾与集成测试固化
2. 市场安装异步化与补偿
3. 管理端统计与审计接口
4. 计划步骤记录
5. 长任务状态管理
